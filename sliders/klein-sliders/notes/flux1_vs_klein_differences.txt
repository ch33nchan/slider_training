flux1-dev vs flux2-klein — notes on what actually changes for slider training


the core slider idea stays the same. you're still computing gt_pred = target_pred + eta * (positive_pred - negative_pred) and backpropping through the LoRA. none of that touches the architecture so it carries over fine.

the first thing that actually matters is the text encoder. flux1-dev uses two of them — clip and t5-xxl — and you need outputs from both, one for the pooled embedding and one for the full sequence. klein throws all of that out and uses a single qwen3 model. the trick is you don't use the final layer output, you pull hidden states from three intermediate layers (9, 18, 27), concatenate them along the hidden dimension, and that's your encoder_hidden_states. there's no pooled projection at all. the transformer forward call simply doesn't have that argument.

on the transformer side, flux1-dev uses FluxTransformer2DModel and the attention layers inside it are just called Attention in diffusers. klein uses Flux2Transformer2DModel and its attention modules are Flux2Attention and Flux2ParallelSelfAttention. the LoRA code needs both names in its target list or the 80 modules it's supposed to wrap won't be found. also the path names of those modules use transformer_blocks.N.attn style naming, not the unet-style up_block/down_block/mid_block paths that the old train method filters were built around. this meant the xattn-up / xattn-down / xattn-mid filters were useless for klein. we just support xattn (all attention) and full (all layers) and that's enough.

classifier-free guidance is handled very differently. flux1-dev doubles the batch — runs the same noisy latent twice, once conditioned and once not, then combines. klein takes the guidance scale as a scalar tensor passed directly into the transformer, no batch doubling. there's a config flag called guidance_embeds on the transformer that tells you whether this is expected. on the undistilled base model it was False when we actually loaded it and checked, meaning you just don't pass guidance at all and the forward call ignores it entirely.

the latent geometry is where the real surprises were. the writeup i had originally said it was the same as flux1 — same 16-channel vae, same 3D position ids, same _unpack_latents. this was wrong. klein's vae is AutoencoderKLFlux2, 32 latent channels. the position ids are 4D not 3D — the original code was using torch.zeros(..., 3) everywhere and that caused an out-of-bounds cuda assert immediately on first run. changing everything to torch.zeros(..., 4) fixed it. the decode path is also different — you can't just call _unpack_latents and then vae.decode. klein does _unpack_latents_with_ids (a position-ID scatter, not a simple reshape), then batch norm denormalization (latents * bn_std + bn_mean), then _unpatchify_latents, then vae.decode. trying to skip any of those steps produces broken red-noise output. the right solution was to stop using a custom pipeline entirely and just use Flux2KleinPipeline from diffusers which handles all of this correctly.

the scheduler is also different in one important way. FlowMatchEulerDiscreteScheduler for klein uses dynamic shifting, meaning set_timesteps requires a mu parameter computed from the image sequence length. calling it without mu raises an error immediately. mu is just a linear function of seq_len computed from four config values (base_image_seq_len, max_image_seq_len, base_shift, max_shift). for 512x512 images with vae_scale=16 the packed sequence length is 256 tokens and mu comes out around 0.5. the scheduler is also stateful — it advances an internal step index on every .step() call — so if you reuse the same scheduler object across training iterations without calling set_timesteps again it runs off the end of the sigmas array on the second or third step.

one thing that does not change at all is the LoRA mechanism. the LoRAModule replaces the linear layer forward directly via apply_to(), and the context manager just sets multiplier to nonzero to activate it. this means for inference you don't need to monkey-patch anything — you just do with network: pipe(...) and the LoRA is live for the full denoising pass.

the distilled vs undistilled model thing matters the same way it does for flux1. you want the base undistilled checkpoint for training. FLUX.2-klein-base-4B is the right one. the standard release is distilled to 4 steps and the training signal is gone.
